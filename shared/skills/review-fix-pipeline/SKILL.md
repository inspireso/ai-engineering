# Review-Fix Pipeline — 审查修复闭环

标准化代码审查与修复流程：先审查列出问题，逐项修复后全量测试验证。

---

## Step 1: 审查清单

对当前变更（或指定文件）进行代码审查，以中文输出，按三级分类：

```
## 审查结果

### 🔴 严重（必需修复）
安全漏洞、数据丢失风险、空指针、事务错乱

### 🟡 重要（建议修复）
性能问题、边界值缺失、异常处理不当、日志遗漏

### 🟢 建议（可选优化）
代码风格、可读性、命名建议、注释多余或缺失
```

每个问题应标明：**文件路径:行号** + 问题描述 + 建议修改方式。

**审查维度：**
- 逻辑正确性 → 条件判断、循环、状态机是否正确
- 安全性 → SQL 注入、XSS、权限绕过、敏感数据泄露
- 异常处理 → catch 是否合理、是否吞异常、finally 是否缺失
- 空指针风险 → 所有引用前是否有 null 检查
- 边界值 → 空集合、null 入参、最大值、特殊字符
- 测试覆盖 → 是否有测试覆盖变更逻辑

## Step 2: 确认优先级

向用户展示问题清单，AskUserQuestion 确认优先级：

```
请确认修复优先级：
A) 全部修复（推荐）
B) 仅修复严重 + 重要问题
C) 仅修复严重问题
D) 自定义选择
```

根据用户选择确定修复范围。

## Step 3: 逐项修复

逐项修复，从严重到建议。每修复一个问题：

1. 读取问题文件和上下文
2. 应用修复
3. 在问题列表中标注状态 `✅ 已修复` 或 `⏭️ 已跳过`

跨文件修改前，先使用 Grep 搜索所有引用点，避免遗漏。

## Step 4: 全量测试验证

```bash
# 运行所有相关测试
./mvnw test -pl <模块名> 2>&1 | tail -30
```

若测试失败：
- 分析根因是修复引入的问题还是原有测试环境问题
- 修复后重新运行
- 最多 3 轮修复循环，仍不过则 AskUserQuestion 升级给用户

## Step 5: 报告汇总

输出最终修复汇总：

```
## 修复汇总

总问题数: 8
- 🔴 严重: 3 / 3 已修复 ✅
- 🟡 重要: 4 / 4 已修复 ✅
- 🟢 建议: 1 / 1 已跳过 ⏭️

测试结果: 37/37 通过 ✅
涉及文件: 9
```
