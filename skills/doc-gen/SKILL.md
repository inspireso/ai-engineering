# Doc Gen — 先大纲后生成文档

生成大型文档前，先探索代码结构输出大纲确认，再展开撰写。

---

## Step 1: 探索代码结构

快速了解要文档化的模块结构：

```bash
# 列出模块目录结构（限 3 层深度）
find src/main/java/<模块包路径> -maxdepth 3 -type f -name "*.java" | head -30

# 查看关键类
ls src/main/java/<模块包路径>/*.java

# 查看配置文件（如有）
ls src/main/resources/ 2>/dev/null
```

## Step 1b: 探索测试用例

通过测试深入了解模块的预期行为、典型用法和边界情况：

```bash
# 查找对应的测试目录
find src/test/java/<模块包路径> -maxdepth 3 -type f -name "*.java" | head -20

# 列出关键测试类
ls src/test/java/<模块包路径>/*.java 2>/dev/null
```

分析测试文件中的关键信息：

- **测试场景：** 覆盖了哪些功能路径？哪些是核心用例、哪些是边界情况？
- **典型用法：** 测试中是如何构造对象、调用方法的？可作为"使用示例"的参考
- **异常路径：** 测试覆盖了哪些异常和错误处理？帮助理解模块的健壮性设计
- **Mock 与依赖：** 测试使用了哪些 mock 和外部依赖？反映模块的依赖关系
- **命名风格：** `given_when_then` / `should` 等约定，反映团队规范

在输出大纲时一并说明从测试中获得的洞察（如"测试覆盖了 X 个场景，核心流程为..."）。

## Step 2: 输出大纲

输出文档大纲供用户确认：

```
## 文档大纲：[文档标题]

根据代码探索，建议文档结构如下：

1. **概述**
   - 模块定位和职责
   - 核心能力

2. **核心架构**
   - 主要类和接口
   - 关键流程（时序或流程图）

3. **使用示例**
   - 典型用法
   - 配置方式

4. **注意事项**
   - 已知约束
   - 常见问题

---
以上结构是否符合预期？是否有需要增删改的章节？
```

使用 AskUserQuestion 确认或让用户补充需求。

## Step 3: 展开撰写

根据确认后的大纲逐节撰写。每完成一节，读取关键源代码确保内容准确。

文档风格要求：
- 使用中文撰写
- 语言简洁，避免冗余
- 关键类和接口标注包路径
- 代码示例使用正确的语法高亮
- 架构图使用 Mermaid 格式

## Step 4: 输出确认

完成草稿后，让用户做最终确认：

```
文档已生成：[文件路径]

简检查项：
- [ ] 核信息准确
- [ ] 代码示例可编译/运行
- [ ] 架构描述与实际一致
- [ ] 格式正确

是否需要调整？
```
